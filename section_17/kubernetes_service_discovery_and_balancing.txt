Section 17. Server-side discovery and load balancing using Kubernetes
1. Introduction
So far i'm using Server-side discovery and load balancing using client side discovery with Eureka server.

1.1 Client-side Service Discovery (e.g., using Eureka Server):

Advantages:

Offers control over load balancing strategies since the client chooses which instance to communicate with.
Can be more transparent in how requests are routed.

Disadvantages:

Requires manual maintenance of the Eureka Server.
Developers need to configure each microservice to connect to the Eureka Server, increasing overhead.

1.2 Server-side Service Discovery (e.g., using Kubernetes):

Advantages:

Reduced developer overhead since microservices do not need to register themselves manually and send future heartbeats.
Simplified configurations, allowing microservices to send requests directly using service names.
Load balancing is managed automatically within the Kubernetes cluster, enabling developers to focus on application logic.

Disadvantages:

Less control over load balancing strategies, which are determined by the Kubernetes cluster.

1.3 Both service discovery approaches have their pros and cons, and the choice should depend on specific business needs.
1.4 See slide from page 215 for details about how the Service Discovery and Load Balancing works using Kubernetes:
The K8s discovery server is responsible to monitor the application instances and maintaining the details of them.
When a microservice needs to communicate with a backing service, it simply invokes the service URL exposed by the K8s' service.
The K8s Service will take care of loadbalancing the requests at the server layer. So clients are free from load balancing in this scenario.
