7. Test Kubernetes self healing
Update the accounts.yml Kubernetes config file to have 2 replicas and apply it with the command.
spec:
  replicas: 2
This will make K8s to maintain 2 replicas working of the account container.
Commands:
kubectl get replicaset # check the desired config about the container
kubectl get pods # before updating the replica to 2 there is one pod for each Microservice
kubectl apply -f 5_accounts.yml
kubectl get pods # check account will have now 2 pods
kubectl delete pod accounts-deployment-858bf48958-h4j5z # this delete a pod. k8s will create a new one instanly
kubectl get pods # check the new pod is created to have 2 pods or replica as specified in the config file.
kubectl get events --sort-by=.metadata.creationTimestamp # see the events sorted. when a pod was deleted and new one created.

Question here is how K8s realizes the pod is not healty and it decides to create a new one? In docker compose there is a health command. I don't know in K8s.

8. Deploying to a Kubernetes cluster without downtime, if your face any issue you can rollback to previous state without manual effort.
kubectl scale deployment accounts-deployment --replicas=1 # similarly to the config file and aplying it, this will set the account deployment replica to 1.
kubectl describe pod pod-id # shows data about the pod like the docker image it used to create the pod.
kubectl set image deployment gatewayserver-deployment gatewayserver=eazybytes/gatewayserver:s11 --record # update using this command the pod's docker image. s11 doesn't have the security layer of keycloak. if the image doesn't exist or there is any issue with it K8s wont remove the previous pod.
kubectl get events --sort-by=.metadata.creationTimestamp # see the event of K8s pulling the image and scaling it up.
kubectl rollout history deployment gatewayserver-deployment # see the revision history to rollback 
kubectl rollout undo deployment gatewayserver-deployment --to-revision=1 # specify the revision to rollback

Kubernetes is capable of autoscaling the container automatically based upon some defined requirements. See documentation:
https://kubernetes.io/docs/concepts/workloads/autoscaling/horizontal-pod-autoscale/
to autoscale for example based on CPU or memory usage.

9. Update service type to ClusterIP to not expose other MS than the edge MS (gateway-server)
ClusterIP # Kubernetes has a load balancer and it send the request to the right working node where the MS is.
LoadBalancer # Kubernetes exposes the service to an IP and port. Ex:  localhost 8080:32124/TCP -> IP is localhost and listening external call through 8080 and the node has the port 32124 which can change when pods get recreated
NodePort # not recommended, it's to expose the working node directly which can change and the client need to have the updated port.
Command:
kubectl get services # to see the services
